# Forms

`@flexi-ui/form` is a thin wrapper around React Aria Components' `Form` primitive. It speaks the browser's native constraint-validation API, so `isRequired`, `type="email"`, `pattern`, `minLength`, and `maxLength` just work — and FlexiUI's `Input` automatically renders the resulting error messages.

## Minimal form

```tsx
import { Form } from '@flexi-ui/form'
import { Input } from '@flexi-ui/input'
import { Button } from '@flexi-ui/button'

export function SignIn() {
  return (
    <Form
      onSubmit={(e) => {
        e.preventDefault()
        const data = new FormData(e.currentTarget)
        console.log(Object.fromEntries(data))
      }}
    >
      <Input label="Email" name="email" type="email" isRequired />
      <Input label="Password" name="password" type="password" isRequired />
      <Button type="submit">Sign in</Button>
    </Form>
  )
}
```

`Form` prevents submission if any field is invalid and surfaces the browser's validation message next to the offending field.

## Validation rules

`Input` forwards every native validation attribute straight to the underlying `<input>`:

```tsx
<Input
  label="Username"
  name="username"
  isRequired
  minLength={3}
  maxLength={20}
  pattern="[a-z0-9-]+"
  description="Lowercase letters, numbers, and hyphens."
/>
```

The browser checks the value on blur/submit and exposes the failure reason via the constraint-validation API. React Aria reads that, and `Input` renders it inside the error slot — no wiring required.

## Custom error messages

Override the browser-generated copy for a single field by passing `errorMessage` and flipping `isInvalid`. This is the pattern you want when the server reports a failure:

```tsx
const [usernameTaken, setUsernameTaken] = useState(false)

<Input
  label="Username"
  name="username"
  isInvalid={usernameTaken}
  errorMessage={usernameTaken ? 'That username is already taken.' : undefined}
/>
```

The same slot is used whether the error comes from native validation or your own state, so transitions between them are seamless.

## Server-side validation errors

`Form` accepts a `validationErrors` object that maps field `name` to an error string. Use it after a failed server action:

```tsx
const [errors, setErrors] = useState<Record<string, string>>({})

async function handleSubmit(event: React.FormEvent<HTMLFormElement>) {
  event.preventDefault()
  const result = await signUp(new FormData(event.currentTarget))
  if (!result.ok) setErrors(result.errors)
}

return (
  <Form onSubmit={handleSubmit} validationErrors={errors}>
    <Input label="Email" name="email" type="email" isRequired />
    <Input label="Password" name="password" type="password" isRequired />
    <Button type="submit">Create account</Button>
  </Form>
)
```

Every field whose `name` appears in `validationErrors` renders the message automatically.

## Controlled values

`Input` supports controlled (`value` + `onChange`) and uncontrolled (`defaultValue`) modes — same as a native input:

```tsx
const [email, setEmail] = useState('')

<Input
  label="Email"
  type="email"
  value={email}
  onChange={setEmail}
/>
```

Controlled mode is what you want when you're reacting to input (live search, inline previews). Uncontrolled is what you want when the `FormData` of submission is the only thing you care about.

## Real-world pattern: async username check

```tsx
const [username, setUsername] = useState('')
const [checking, setChecking] = useState(false)
const [error, setError] = useState<string | null>(null)

async function validate(value: string) {
  if (!value) return
  setChecking(true)
  const { ok } = await fetch(`/api/username?u=${value}`).then((r) => r.json())
  setChecking(false)
  setError(ok ? null : 'That username is already taken.')
}

return (
  <Input
    label="Username"
    value={username}
    onChange={setUsername}
    onBlur={() => validate(username)}
    isInvalid={!!error}
    errorMessage={error ?? undefined}
    description={checking ? 'Checking…' : undefined}
  />
)
```

## Accessibility

- Labels are associated with inputs via React Aria — no manual `htmlFor` / `id` dance.
- Error messages use `aria-describedby` so screen readers announce them when focus reaches the invalid field.
- Prefer native `type` values (`email`, `tel`, `url`, `number`) so mobile keyboards and autofill behave correctly.
- For fields without a visible label, pass `aria-label` — never omit the label entirely.
